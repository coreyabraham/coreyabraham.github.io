Project Details

Game Engine: Unity

Platform: PC // Windows 7+ x64

Role(s): Systems and UI Programmer

Development Date: 10th of June, 2024 - 15th of July, 2024

Brief

Monstacking is a 3D stacking game designed with Virtual Reality in mind as you grab oncoming vehicles in an attempt to stack them on top of a recently lodged bridge and gain a highscore! The game in question is fairly simple as it was only in development for approximately 4 weeks. The game would've had more time to be developed seeing as there was two weeks within that time to begin with, however, they were reserved for holidays, so realistically it was a four week project.

Contributions

My contribution to the project came in the form of Programming the entire project, as we were part of a small group that only had the one Programmer in it, that being myself of course! Most of the project's runtime on my behalf was used to experiment and figure out how to get the Virtual Reality kit Unity provides via it's Package Manager works, of course that being the XR Device Simulator package.

A more notable use of time when it comes to the project comes from the use of Data File Management. I wrote a `DataManager.cs` C# Class for the project to allow tracking of Highscores to give Players more insentive to replay the game instead of it being a one-and-done experience! It nearly didn't make the cut however due to UI issues at the time of the project wrapping up it's development and the Script as a whole not working when deployed to a real Android XR system without explaination. After a really long time debugging the problem however, it turns out that I had setup the Script wrong when it comes to the Directory it reads and writes to. It originally wrote to a Directory it creates on startup (if it isn't already present within the filesystem) to write the data within that was stored where the game's executable was. However, after doing some research, it turns out that writing to that Directory, let alone creating it with XR systems, does NOT have permissions to write there. So I ended up using the persistant location option Unity has by default and just like that, the system made the cut and was functional in the final build like nothing happened!

Outside of that however, there really isn't all that much to talk about with this project aside from the smaller details. All User Interface elements (aside from the in-game billboard) were programmed right before the project's development had concluded. Mainly due to how fairly simple it was to implement and how much depth we decided to go into with it later on. We knew as a collective team we wanted a User Interface for the game, however only at the beginning and end, as having it during gameplay non-diegetically would've likely ruined the immersion the game was going for.
